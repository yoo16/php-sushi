# Frontend Build Skill

このファイルは、Codex がこのプロジェクトで `Vite + React` フロントエンドを実装するときの作業ルールを定義する。

## ミッション

既存の PHP アプリに対して、注文画面を中心に React フロントエンドを導入する。目的は SPA 化そのものではなく、既存 API を活かしつつフロント実装の保守性と開発速度を上げること。

## 最初に確認すること

1. `docs/frontend/SPEC.md` を読む
2. `app/views/home/menu.php` を読み、PHP から渡される値を確認する
3. `js/app.js` を読み、現行機能を洗い出す
4. `api/` 配下のレスポンス形式を崩さないことを確認する

## 実装ルール

- 既存の PHP ルーティングは維持する
- まずは注文画面だけを React 化する
- 既存 API はできるだけそのまま使う
- API 仕様を変える場合は、先に `SPEC.md` を更新する
- ディレクトリ構成は `frontend/` を起点に整理する
- コンポーネント分割は画面責務ベースで行う
- 状態管理ライブラリは、必要になるまでは導入しない
- フォームやモーダルは見た目より動作整合性を優先する

## 推奨実装順

1. `frontend/` を Vite + React で初期化する
2. エントリポイントと `App` を作る
3. PHP テンプレートに mount 要素を置く
4. 初期データを PHP から React へ渡す
5. カテゴリ、商品、注文の取得処理を service 化する
6. 一覧 UI とモーダル UI を React に置き換える
7. 会計処理をつなぐ
8. ビルド成果物を PHP から読む

## 触る可能性が高いファイル

- `app/views/home/menu.php`
- `app/views/components/head.php`
- `js/app.js`
- `api/category/fetch.php`
- `api/product/fetch.php`
- `api/order/fetch.php`
- `api/order/add.php`
- `api/order/billed.php`

## 変更時の判断基準

- 既存挙動を再現するための変更か
- React 導入に必要な最小変更か
- PHP 側との境界が明確になっているか
- 開発時と本番時の読み込み経路が説明可能か

## 避けること

- 最初から管理画面まで React 化する
- API と画面を同時に大きく作り替える
- グローバル状態を先回りで複雑にする
- CSS 設計を先に大きく抽象化する

## 完了時チェック

- `SPEC.md` の完了条件を満たしている
- PHP から必要な初期値が受け渡せている
- React 側で注文導線が一通り動く
- ビルド成果物の読み込み経路が明確
- 不要になった旧コードの扱いが整理されている
